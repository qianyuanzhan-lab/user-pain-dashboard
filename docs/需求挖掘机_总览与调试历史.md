## 一、产品定位

### 1.1 要解决的问题

中国互联网有海量真实用户吐槽散落在 **App Store 评论 / Google Play 评论 / 技术社区**，但产品团队很难高效从里面挖出「下一步该做什么」。

传统做法要么人肉读几千条（贵、慢），要么靠情感分析打个分（浅）。

**需求挖掘机** 的目标是：**自动化地把 18,000+ 条用户评论压缩成 300 个可执行的"需求点"**，并按赛道、优先级、AI 可介入程度排序，配上原始证据和跨产品分布。

### 1.2 覆盖赛道

| 赛道 | 目标产品 | 候选池规模 |
|------|----------|---:|
| **wechat** 微信生态 | 微信 App | 800 条（只用 App Store + Google Play）|
| **social** 社交娱乐 | 抖音、小红书、Soul、微博、Discord… | 2,116 条 |
| **ai** AI 工具 | ChatGPT、DeepSeek、Kimi、Claude、豆包… | 2,980 条 |
| **more** 效率工具 | 滴答清单、Obsidian、Notion… | 1,365 条 |

### 1.3 最终产出

- **Dashboard 前端**：分赛道展示 300 个需求点，每个点带标题、描述、提及数、跨产品分布、典型用户原声、AI 介入方案
- **排序 / 打标**：按综合分排序（重要度 30% + AI 参与度 40% + 共性 30%）；类型三分法：功能缺失 / 体验问题 / 平台策略
- **标识**：🟢 跨产品通用 · 🔴 本周新增

## 二、技术栈

### 2.1 核心技术选型

| 层级 | 技术 | 为什么 |
|------|------|--------|
| **抓取** | Python + 各平台 API/RSS | App Store iTunes RSS、Google Play 官方爬虫、HN Algolia API |
| **翻译** | DeepSeek API | HN 是英文，翻成中文再处理；¥0.5-1/千条 |
| **抽原子** | DeepSeek API | 让 LLM 从每条评论里抽出"结构化需求原子"（need_statement/sentiment/perspective/demand_type）|
| **聚类** | 本地 embedding `BAAI/bge-small-zh-v1.5` + sklearn 层次聚类 | 不花 API 成本、几千条 10 秒；中文语义准 |
| **打标+合并** | DeepSeek API | 给每簇起标题、demand_type、demand_tier、AI 评分；再做二级合并 |
| **描述修订** | DeepSeek API | 修订 description 让它紧扣 title 且扎根原始用户诉求 |
| **Dashboard 前端** | React + Vite + TypeScript + Tailwind | 快速开发；JSON 直接 import；部署即静态文件 |
| **自动化** | GitHub Actions cron + GitHub Pages | 周一自动跑，免费无限 |

## 三、十步管线（pipeline）

这是系统的灵魂，从原始评论到最终需求点共 10 步：

```
[Step 0]  crawl_incremental.py        增量抓取（近 14 天 × 三源 × 四赛道）
    ↓
[Step 1-4] pipeline_step1_4.py        清洗 + 过滤 + 翻译合并
           ·  去重 / 按时间裁剪 / 关键词过滤 / 去低质
           ·  HN 主题相关性过滤（微信赛道不用 HN）
    ↓
[Step 5]  pipeline_step5_atoms.py     LLM 抽需求原子
           ·  batch 20 并发 + 断点续跑
           ·  产出: need_statement, sentiment, perspective, demand_type
    ↓
[Step 6]  pipeline_step6_cluster.py   本地 embedding 聚类
           ·  distance_threshold=0.42（调过多次）
    ↓
[Step 7]  pipeline_step7_label.py     LLM 打标
           ·  每簇生成 title / description / demand_type / ai_score
           ·  prompt 含三分法判定流程（平台策略 > 功能缺失 > 体验问题）
    ↓
[Step 8]  pipeline_step8_merge.py     二级合并（语义相似簇）
           ·  sim 阈值 0.82
    ↓
[Step 8.5] pipeline_step8_5_dedup.py  LLM 全局去重（按需启用）
           ·  分批 30 个 + 交叉批，严格判定
           ·  wechat/ai 不跑（避免过度合并）
    ↓
[Step 8.6] pipeline_step8_6_refine.py 描述修订
           ·  每点拿 title + 代表诉求 → LLM 重写 description
           ·  原则: 紧扣 title + 扎根证据
    ↓
[Step 9]  pipeline_step9_dashboard.py 导出 Dashboard JSON
           ·  综合分排序（30/40/30）
           ·  evidence_samples 挑 3-5 条最强证据
    ↓
[Step 10] pipeline_step10_merge_new.py 增量合并（只自动化模式用）
           ·  新需求与旧主数据 sim 对比
           ·  sim>0.85 并入旧；否则打 is_new + discovered_at
    ↓
Vite 构建 → GitHub Pages → 公网访问
```

### 3.1 关键参数表

| 参数 | 默认值 | 作用 | 实验记录 |
|------|------|------|------|
| `Step 6 distance-threshold` | **0.42** | 聚类松紧 | 0.35 太细（199 簇），0.50 太粗（60% 并一簇），0.42 刚好（99 簇）|
| `Step 8 sim` | **0.75~0.82** | 二级合并 | social 用 0.75；wechat/ai 最终用 0.82（避免合过头）|
| `Step 8.5 开关` | 按赛道 | LLM 全局去重 | social 有效（175→30），wechat 过激（28→8）→ 默认不跑 |
| `Step 10 SIM_THRESHOLD` | **0.85** | 新旧是否判同 | 高于此阈值并入旧；否则新增打 is_new |
| `综合分权重` | 30/40/30 | 排序 | 经用户选择 B：AI 参与度权重最高 |

### 3.2 综合分公式

```
final_score = 0.30 × importance + 0.40 × ai_relevance + 0.30 × commonality

importance   = log(mention_count) + 评分强度 + 情绪强度 + T1 加成       (0-10)
ai_relevance = ai_score（≥7 时额外 +0.5）                              (0-10)
commonality  = log(产品数) + 跨数据源数                                 (0-10)
```

### 3.3 demand_type 三分法（非常关键）

按顺序问：

1. **平台策略**？ 诉求涉及定价/封号/广告频次/实名/合规/审核规则等商业决策 → ✅
2. **功能缺失**？ 产品没有、用户要新增/支持/开放 X → ✅
3. **体验问题**？ 产品已有但做得不好（慢/乱/不准/不稳/卡）→ ✅

边界案例：
- "推荐不准" = 体验问题（算法已有）
- "希望能关闭推荐" = 功能缺失（需新增开关）
- "广告太多" = 平台策略（频次是决策）
- "广告加载卡" = 体验问题（性能）
- "希望有人工客服" = 功能缺失
- "客服没人回复" = 体验问题

## 五、调试历史（值得记住的坑）

### 5.1 排序演进
- **v1**：按优先级 P0/P1/P2 → 用户反馈"T1 里 AI 分高的没排前面"
- **v2**：用户选权重 30/40/30 → 重要度 + AI 参与度 + 共性
- **结论**：AI 产品场景下，"AI 能介入多少"比"问题多严重"更值得优先看

### 5.2 颗粒度演进（最多反复的部分）
- **v1 (social)**：175 个需求点 → 用户"太细，重复"
- **v2**：加 Step 8.5 全局去重 → 30 个 ✓
- **v3 (wechat)**：用 Step 8.5 → 8 个 ✗ 用户"太粗"
- **v4 (wechat)**：跳过 Step 8.5 + Step 8 sim=0.75 → 28 个 ✓
- **教训**：**Step 8.5 不能一视同仁**，候选池小的赛道要跳过

### 5.3 wechat 噪声问题
- **症状**：微信赛道出现"苹果键盘 / macOS 问题"
- **根因**：HN 数据里大量"讨论其他话题时顺带提 WeChat"
- **修复 v1**：加 HN 关键词过滤 → 还剩 76 条噪声
- **修复 v2**：**微信赛道彻底不用 HN**（HN 是技术视角，不是本地用户）

### 5.4 "日历产品混簇"问题
- **症状**：滴答清单需求点里出现 Outlook / Thunderbird / Wine 样本
- **根因**：embedding 把"日历/日程"做强信号，不同产品的日历话题被聚一起
- **修复**：前端 post-filter
  - 硬过滤（隐藏）：主产品 <40% + 产品数 >6
  - 软标记（绿标）：主产品 <70% + 产品数 ≥3

### 5.5 描述不自洽问题
- **症状**：标题说"Character.AI 翻译支持"，描述/原声说"停止添加无用功能"
- **根因**：Step 8 合并时 title 是 LLM 新起的，但 description/representative 取了某簇的原始诉求 → 对不上
- **修复**：新增 Step 8.6 让 LLM 根据 title + 原始证据重写 description
- **原则**：description 必须解释 title，且扎根于实际用户诉求，不能凭空

### 5.6 "要点 / 用户建议"UI
- **v1**：弹窗每条样本下方显示小徽标 + "XX 应该…"
- **用户反馈**：句子太乱，看不清每条在讲啥
- **v2**：做成顶部一整条浅蓝背景 + 字段"要点"
- **v3**：把"XX应该/希望"句首规范成"用户建议"
- **标签文字保留"要点"** + **内容开头"用户建议"**

### 5.7 类型分类混乱
- **v1**：用 sentiment（negative/mixed/positive）映射 UI 类型标签 → 张冠李戴
- **v2**：直接用后端 demand_type 字段 → 功能缺失/体验问题/平台策略
- **v3**：Step 7 prompt 加详细三分法判定流程 + 边界案例，四赛道统一重打标

## 六、成本核算

单次全量四赛道：
- **DeepSeek API**: ¥8-10（Step 5 翻译抽原子占 70%）
- **时间**: 30-40 分钟（Step 5 是瓶颈，~3,000 条 × 5 分钟 = 15 分钟/赛道）
- **Embedding**: 本地跑，零 API 成本

GitHub Actions：
- Public repo 免费无限
- 每周跑一次 × 60 分钟 ≈ 240 min/月

**月度总成本**：¥40（API）+ 0（CI）+ 0（Pages）= **¥40/月**
